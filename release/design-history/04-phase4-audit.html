Phase 4 Audit

Scripted opponents · FC-AU-003 · audit of sovereign-solo-phase4-scripted-opponents.html

Twelve checks verified against the source. Phase 4 successfully migrates Phase 3's solo state machine to a 3-player table with two deterministic profiles, auctions, and multi-player rent. Two critical bugs, three high/medium bugs, and one cleanup found.

§ 01Verdict

Pass with fixes

Functional. Two critical patches block freeze; three additional fixes recommended before Phase 5.

Card-driven movement (Yellow Fever) and card-driven auctions (Speculation Fever) are both broken by the same root cause: finishLanding() unconditionally ends the turn after a card resolves, clobbering any sub-phase the card set up (auction) or any deferred landing-resolution flag (movement). These are the freeze blockers. The remaining issues are: failed Acts don't actually re-queue (ledger claims they will), opponent decideBuy / decideAuctionBid reasons are dropped at the BUY/DECLINE_ASSET boundary, Crisis-exit ignores the doubles/skip options, and Strict Construction's opponent choice is hardcoded instead of profile-driven.

§ 03Bugs Found

BUG-01 · Critical · Card-driven movement (Yellow Fever) skips landing resolution

Source: finishLanding() and Market Shock card #8 "Yellow Fever Outbreak". RESOLVE_CARD calls finishLanding(s) which unconditionally calls END_TURN. Yellow Fever sets s.pendingResolveLanding = true intending to trigger a follow-up RESOLVE_LANDING for the new position. But END_TURN advances activePlayerIndex first. The queueMicrotask check then dispatches RESOLVE_LANDING for the next player at whatever space they're standing on.
Land on a Market Shock space, draw Yellow Fever (card #8). Player moves to nearest Commercial Infrastructure. No rent is paid even when that space is owned by an opponent. Next player's turn begins and their turn's landing fires as if they had rolled and landed — but they haven't rolled yet. Ledger shows an out-of-order RESOLVE event.
// In finishLanding(s): function finishLanding(s) { if (s.pendingResolveLanding) { s.pendingResolveLanding = false; return reduce(s, { type:'RESOLVE_LANDING' }); } s.phase = 'awaiting-roll'; return reduce(s, { type:'END_TURN' }); } // Remove the matching pendingResolveLanding check from dispatch() // since finishLanding now handles it inline before END_TURN.

BUG-02 · Critical · Card-driven auction (Speculation Fever) gets clobbered

Source: Market Shock card #3 "Speculation Fever" sets s.pendingAuction and s.phase = 'auction'. Then RESOLVE_CARD calls finishLanding(s), which sets s.phase = 'awaiting-roll' and dispatches END_TURN — destroying the auction's phase before any bids can be collected.
Land on Market Shock, draw Speculation Fever. Track movement (Resistance +1) is logged. Auction is supposed to open for an unowned Rev/State Debt property. Instead the turn ends immediately and the auction is silently dropped. pendingAuction remains set on state but is never rendered or resolved.
// In finishLanding(s), respect card-set sub-phases: function finishLanding(s) { if (s.phase === 'auction') return s; // auction is live, hold if (s.phase === 'card-choice') return s; // sub-decision pending if (s.pendingResolveLanding) { s.pendingResolveLanding = false; return reduce(s, { type:'RESOLVE_LANDING' }); } s.phase = 'awaiting-roll'; return reduce(s, { type:'END_TURN' }); } // And in processNextAuctionBid() resolution path, end the // triggering player's turn explicitly via END_TURN — already done.

BUG-03 · High · Failed Acts do not re-queue · ledger claims otherwise

Source: CAST_VOTE case logs "will re-queue next lap" when an Act fails, but the reducer doesn't modify state.acts beyond clearing current. On the next lap, BEGIN_LAP calls ACTS.find(a => a.lap === s.lap) which returns the canonical Act for that lap — the failed Act is gone.
Vote NO three times on the Funding Act in lap 1. Ledger says "Funding Act fails · will re-queue next lap." Lap 2 begins, Assumption Act is revealed for vote. Funding Act is never seen again. Per Phase 3 spec this was an accepted simplification — but the ledger claim is misleading.

BUG-04 · High · Opponent decision reasons lost at BUY_ASSET / DECLINE_ASSET

Source: renderControls() calls dispatch({type:'BUY_ASSET', reason}) for opponents after computing decideBuy(). The BUY_ASSET reducer case reads s.pendingLanding but ignores action.reason. The opponent's profile reason ("aligned with profile; cash 820 ≥ cost+reserve 420") is dropped; the ledger logs only the generic "Buy Customs House" entry. Same story for DECLINE_ASSET. decideEarlyVoteFee() is similarly never called in the reducer.
Open the ledger after Hamilton (Treasury/Finance) buys an asset. The reason for the buy is not present. The human cannot read the rule that fired without inspecting the source. Spec requirement § 04 says "every decision logs a reason referencing visible state" — partially met for votes / bids, not met for buy / decline / upgrade for opponents.
// In BUY_ASSET: - s = logRow(s, { actor: p.name, event:'OWN', - detail:'Now owns ' + sp.name, cls:'cash' }); + s = logRow(s, { actor: p.name, event:'OWN', + detail:'Now owns ' + sp.name + + (action.reason ? ' · ' + action.reason : ''), + cls:'cash' }); // In DECLINE_ASSET: same pattern. // In UPGRADE_ASSET: append action.reason if provided (already mostly done). // Also call profile.decideEarlyVoteFee(s, pIdx) in runOpponent() // before CAST_VOTE — currently the function is defined but never invoked.

BUG-05 · Medium · Crisis exit ignores doubles and skip-turn options

Source: ROLL_DICE case handles Crisis by auto-paying 50 TN if p.cash >= 50, otherwise skipping. Rules give three options: (a) pay 50 TN, (b) roll doubles, (c) skip one full turn and exit on the following turn. Only (a) is implemented. The human is denied agency; opponents waste cash they could have saved by rolling doubles or skipping.
Land on Constitutional Crisis. Next turn, ROLL_DICE is dispatched. Cash is silently debited 50 TN before the dice are rolled. The human never sees a choice; opponents never roll for doubles.
// Split Crisis exit into a pre-roll sub-phase for the human: // when activePlayer is human and inCrisis, set phase='crisis-choice' // and render a 3-button UI (Pay 50 / Roll for doubles / Skip turn). // For opponents, use a profile-driven decision: roll if cash < 100, // pay if cash ≥ 200, skip otherwise. Each path is deterministic // and writes its reason to the ledger. // Minimal patch for freeze: implement the human UI choice; // opponents continue with the pay-50 fallback (cheap to keep).

BUG-06 · Low · Strict Construction Objection · opponent choice hardcoded

Source: DRAW_CARD case, opponent-path. When an opponent draws a card with choices (only Strict Construction), code dispatches RESOLVE_CARD_CHOICE with choiceIndex: 1 (always "pay 50 TN") via setTimeout. This is profile-blind: Merchant/Infrastructure might prefer Crisis (it preserves cash for the route grand-slam); Treasury/Finance might prefer paying.
Have an opponent land on a Republic Debate space and draw card #1. They always pay 50 TN. Profile asset priorities don't enter the decision.
// Add a decideCardChoice(profile, player, card, state) function // to each profile. Treasury/Finance: prefer paying. Merchant/Infra: // prefer Crisis when cash < 200, else pay. Wire it in DRAW_CARD // opponent-path: + const profile = PROFILES[s.players[pIdx].profile]; + const idx = profile.decideCardChoice + ? profile.decideCardChoice(s, pIdx, card) + : 1; + setTimeout(() => dispatch({type:'RESOLVE_CARD_CHOICE', choiceIndex: idx}), 250);

§ 04Non-Blocking Deviations

  • Opponent auto-resolution timer. Opponent decisions fire via setTimeout in renderControls() at 250 ms. If render() is called repeatedly in fast succession (it isn't, but defensively), multiple timers could stack. Each fires with a guard (if STATE.phase !== 'asset-decision' || STATE.activePlayerIndex === 0 return) so behavior is correct but the pattern is fragile. Recommend moving the auto-resolve to a single queue inside dispatch() rather than render-bound.
  • Failed-Act re-queue. Acknowledged Phase 3 simplification, but the ledger should be corrected (see BUG-03).
  • End-of-game Influence on Default during lap 7. If Default fires after turn 21 (the human's last turn) but before END_TURN advances, the lap-7 increment may double-fire. Verified safe by inspection — pendingDefault is processed before END_TURN — but worth adding a regression test.

§ 05Freeze Verdict

Pass with fixes

Apply BUG-01 and BUG-02 immediately. BUG-03 through BUG-06 before Phase 5 spec lock.

BUG-01 and BUG-02 are blockers: they break two specific cards in a way that produces incorrect game outcomes (missed rent, dropped auctions). The fix is two lines in finishLanding(). BUG-03 is a misleading ledger string with no behavioral consequence — fix in either a five-character text edit or a Phase 5 proper-re-queue implementation. BUG-04 degrades the legibility goal that scripted opponents promised (every decision should be inspectable). BUG-05 reduces human agency in a way the rules explicitly allow. BUG-06 is profile fidelity at the edges.

After BUG-01 and BUG-02 are patched, Phase 4 is functional and freeze-ready. The remaining four bugs do not prevent completing a 3-player game.

§ 06Patch Application Order

  1. BUG-01 first. Move the pendingResolveLanding handling inside finishLanding(); remove the duplicate check from dispatch().
  2. BUG-02 second. Add the phase === 'auction' and phase === 'card-choice' guards at the top of finishLanding().
  3. Smoke test. Replay seed 2026. Walk through one Speculation Fever card and one Yellow Fever card. Confirm: (a) Speculation Fever opens the auction panel, all three players bid, asset transfers correctly; (b) Yellow Fever moves the player and triggers RESOLVE_LANDING at the new position with rent paid to opponent if owed.
  4. BUG-03 fix. One-line ledger text correction.
  5. BUG-04 fix. Append action.reason to BUY/DECLINE/UPGRADE ledger entries; call decideEarlyVoteFee() in runOpponent().
  6. BUG-05 fix. Add the human Crisis-choice UI; opponents can keep the pay-50 fallback.
  7. BUG-06 fix. Add decideCardChoice() hooks per profile.
  8. Freeze. Tag as sovereign-solo-phase4-scripted-opponents.html v0.2. Phase 5 (narration + replay UI + save/load) is unblocked.

§ 07What Phase 5 Inherits
