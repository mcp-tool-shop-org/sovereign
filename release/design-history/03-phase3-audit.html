§ 03Bugs Found · Required Fixes

BUG-01 · Critical · Endgame UI never renders on lap-7 completion

Source: MOVE_TOKEN reducer case, lines 630–644.
When the player passes Treasury Opens on what would be their 8th time across the start line (completing lap 7), PASS_TREASURY_OPENS is dispatched recursively. That handler increments s.lap to 8, calls END_GAME, and END_GAME sets s.phase = 'game-over' and s.status = 'gameOver'. Control returns to MOVE_TOKEN, which then unconditionally executes s.phase = 'resolve-landing' — clobbering the game-over phase. render() checks STATE.phase !== 'game-over' in renderEndgame() and hides the endgame view. The player sees the regular grid with status flagged but no scoring panel.
--- MOVE_TOKEN case +++ MOVE_TOKEN case case 'MOVE_TOKEN': { const from = s.player.position; let to = (from + action.amount) % 40; const passedStart = (from + action.amount) >= 40; s.player.position = to; s = logRow(s, { actor:'You', event:'MOVE', detail: '...', cls:'move' }); if (passedStart && to !== 0) { s = reduce(s, { type:'PASS_TREASURY_OPENS' }); } else if (to === 0) { s = reduce(s, { type:'PASS_TREASURY_OPENS', landed: true }); } - s.phase = 'resolve-landing'; + if (s.status !== 'gameOver') s.phase = 'resolve-landing'; return s; }

BUG-02 · High · Bankruptcy never tracked, never ends the game

Source: player.bankruptLaps declared in initialState() at line 521 but never modified. No BANKRUPT action exists.
Spec acceptance criterion #10 lists three game-termination paths: lap-7 completion, Default firing, and player bankruptcy (cash < 0 with no remaining recoverable value). The current state machine has no bankruptcy detection. The player's cash can go arbitrarily negative through tax spaces, card effects, and Crisis exit costs. computeFinalInfluence() reads s.player.bankruptLaps for the − 1 Influence per lap spent bankrupt deduction (rule line in computeFinalInfluence) but that field is always zero. Effect: a player can play through 7 laps with negative cash and never lose Influence for it; no early-termination path triggers.
// Add to adjustCash() helper, after s.player.cash += delta: + if (s.player.cash < 0 && !s.player.bankruptThisLap) { + s.player.bankruptThisLap = true; + s.player.bankruptLaps += 1; + s = logRow(s, { actor:'System', event:'BANKRUPT', + detail:'Cash below zero · lap ' + s.lap + ' counted as bankrupt', cls:'event' }); + } // Reset the per-lap flag in PASS_TREASURY_OPENS, after lap increment: + s.player.bankruptThisLap = false; // Optionally: add END_GAME early-termination for sustained negative cash: // (e.g. if cash < -500 at end of turn, fire END_GAME with status:'bankrupt')

BUG-03 · Minor · Capacity ≥ 8 endgame bonus over-counts

Source: computeFinalInfluence(), lines ~858–862.
The v0.2 rule reads: "If Industrial Capacity ≥ 8, each Manufactures and Strategic Industry owner gains + 2 Influence." The intended reading is one owner gains +2 per system they hold (i.e. up to +4 if they own properties in both systems). The implementation computes n = countOwnedSys('manufactures') + countOwnedSys('strategic-industry') and adds n × 2 — that scales by the number of properties in those systems, not owners. A player with all 3 Manufactures + all 2 Strategic properties scores +10 instead of +4.
--- computeFinalInfluence, Capacity ≥ 8 block +++ computeFinalInfluence, Capacity ≥ 8 block - if (s.tracks.capacity.value >= 8) { - const n = countOwnedSys(s, 'manufactures') + countOwnedSys(s, 'strategic-industry'); - if (n > 0) add('Capacity ≥ 8 bonus', n + ' Mfg/Strategic owners × 2', n * 2); - } + if (s.tracks.capacity.value >= 8) { + let bonus = 0; + if (countOwnedSys(s, 'manufactures') > 0) bonus += 2; + if (countOwnedSys(s, 'strategic-industry') > 0) bonus += 2; + if (bonus > 0) add('Capacity ≥ 8 bonus', + 'owns Mfg ' + (countOwnedSys(s, 'manufactures') > 0 ? '✓' : '✕') + + ' / Strategic ' + (countOwnedSys(s, 'strategic-industry') > 0 ? '✓' : '✕') + + ' · +2 per system held', bonus); + }

§ 04Deviation Notes · Not Required Fixes

DEV-01 · Spec named 18 actions; reducer has 16 cases

Source: reduce() switch statement, lines 589–824.
The spec lists these named actions as required: BEGIN_LAP · ROLL_DICE · MOVE_TOKEN · PASS_TREASURY_OPENS · RESOLVE_LANDING · BUY_ASSET · DECLINE_ASSET · UPGRADE_ASSET · DRAW_CARD · RESOLVE_CARD · RESOLVE_CARD_CHOICE · CAST_VOTE · TRIGGER_DEFAULT · TRIGGER_REBELLION · END_TURN · END_GAME — that is 16. The spec narrative also lists OPEN_ACT_VOTE, RESOLVE_ACT, and UPDATE_TRACK as separate actions in its action-set introduction but the canonical list (above) is 16. The reducer matches the canonical list exactly.
If the audit's stricter reading (18 actions including OPEN_ACT_VOTE / RESOLVE_ACT / UPDATE_TRACK) is correct, the patch is to split those three behaviors out as discrete dispatched cases:
case 'OPEN_ACT_VOTE': // currently part of BEGIN_LAP case 'RESOLVE_ACT': // currently part of CAST_VOTE's "yes" branch case 'UPDATE_TRACK': // currently inlined as adjustTrack() helper // Behavior is unchanged; splitting just gives each its own ledger pen. // Verdict: optional cleanup, not a blocking bug. Functionality is exhaustive.

Other Deviations (intentional · spec-acknowledged)

  • Failed Acts do not return to bottom of sequence. Phase 3 simplification — the failed Act is skipped this lap and the next lap reveals its own Act. The board game's "rotate failed Act one position back" mechanic is deferred to Phase 4. Ledger logs the simplification.
  • Coinage Act pays 0 TN in solo. Correct: spec says "from each other player" and solo has no other players. Credit + Capacity track movements still apply. Ledger notes this.
  • Auctions are skipped. Declined assets remain unowned. Phase 4 will add the auction mechanic. Speculation Fever card logs the simplification.
  • Trade-based cards default to active-player-only effects. Cabinet Bargain grants the active player 1 Influence and applies the Resistance −1 without a trade partner.

§ 05Freeze Verdict

Pass with fixes

Apply BUG-01, BUG-02, BUG-03 · then Phase 3 is frozen.

BUG-01 is the only true blocker: a player who plays cleanly to lap-7 end does not see the Endgame Report. The fix is a one-line guard. BUG-02 is a quiet correctness bug: bankruptcy never affects scoring, which violates the rules. Two lines to fix. BUG-03 over-rewards Mfg-heavy strategies in the endgame bonus by ~ 6 Influence at the high end. Four-line fix.

Once these three patches are applied, all 12 acceptance criteria pass cleanly. DEV-01 is a documentation / spec-interpretation question rather than a code bug; either the canonical 16-action list in the spec is correct (in which case the implementation already matches) or the narrative's 18-action reading is correct (in which case a cleanup pass splits three behaviors into discrete cases without changing behavior). Recommend treating as a follow-up question for the spec author, not a freeze blocker.

§ 06Patch Application Order

  1. BUG-01 first. Guard the s.phase = 'resolve-landing' assignment in MOVE_TOKEN with a status !== 'gameOver' check.
  2. BUG-02 second. Add bankruptcy detection inside adjustCash(); reset the per-lap flag in PASS_TREASURY_OPENS. Optional: add a hard game-end at sustained negative cash.
  3. BUG-03 third. Rewrite the Capacity ≥ 8 block in computeFinalInfluence() to grant + 2 per system held, not per property.
  4. Smoke test. Replay seed 2026 with all three patches; confirm the lap-7 endgame fires, the Influence breakdown matches the new Capacity formula, and a forced-bankruptcy scenario (decline all assets, take Federal Excise three times) shows a non-zero bankruptLaps deduction at scoring.
  5. Freeze. Tag the file as sovereign-solo-phase3-state-machine.html v0.2. Phase 4 (scripted opponents) is unblocked.

§ 07What Phase 4 Inherits

  • State model. The pseudo-schema from the concept doc is implemented and stable. Phase 4 adds a players[] array and converts the singular state.player into the human entry.
  • Reducer pattern. All 16 cases are pure functions on state. Phase 4 adds an OPPONENT_TURN driver and per-profile decision functions.
  • Cards' multi-player branches. The "solo: no opponent" notes mark exactly where Phase 4 must wire in real opponent payouts. Those notes are searchable strings, making the migration mechanical.
  • Deterministic replay. The seed pill + ledger contract is the foundation for the Replay mode in Phase 5. Phase 4 must preserve it: opponent decisions must be deterministic functions of (rng, state).
