Three-tier failure hierarchy now visible C

Default's role is now well-defined. Before v0.18, the failure system had two tiers: Default (Credit 0, catastrophic) and Rebellion (Resistance 12, catastrophic). Both threshold events are far from typical canonical play. v0.18 adds a third tier between "live game state" and "catastrophe":
  • Credit Crisis (Credit ≤ 4, soft) — warning state, +1 Resistance, no reset
  • Rebellion (Resistance 12, catastrophic) — destroys Revenue upgrades, resets to 6
  • Default (Credit 0, catastrophic) — 50 % cash loss per player + 1 upgrade lost, resets to 3
This is the cleanest product shape so far: a visible warning for the failure system without forcing Default itself to carry every level of distress.

Default suppression — structural proof I

Default and Credit Crisis cannot fire on the same credit transition. The trigger condition includes a next > 0 guard: 
if (key === 'credit' && next <= 4 && next > 0 && !s.flags.creditCrisisFired) {
  s.pendingCreditCrisis = true;
}
At next === 0: pendingDefault is set (by the existing line), pendingCreditCrisis is NOT set. The main-loop priority then dispatches TRIGGER_DEFAULT first. TRIGGER_DEFAULT directly assigns s.tracks.credit.value = 3, bypassing adjustTrack — so the creditCrisisFired flag stays unset. If a subsequent card-driven dip puts credit back in the 1-4 warning band, Crisis fires normally.

Interpretation — answering the design question J

The question v0.18 was designed to answer: Can the failure system have a visible intermediate warning state without weakening Default's catastrophic endpoint?

Yes. v0.18 produces:

  • 2 canonical games with a visible CREDIT_CRISIS System event in the ledger, at the precise moments credit drops below 5
  • +1 shared Resistance as the penalty — cross-track interaction with the Rebellion timer, but small enough that no game reaches the Rebellion threshold
  • Default at Credit 0 unchanged in every metric — same fires (0 / 400), same trigger condition, same reducer behavior
  • Balance bands held byte-perfectly (60.0 / 23.5 / 16.5, identical to v0.17)

The product framing this earns: Default at Credit 0 is now correctly understood as a catastrophic collapse condition, not a balance target. The game's failure system now has a working three-tier hierarchy. The conceptual question "should Default be lowered to Credit ≤ 2 or ≤ 4?" can be retired — Credit Crisis fills that role with the right severity, and Default stays as the dramatic endpoint.

Suggested v0.19 directions (if authorized — observation only)

  • v0.19 — broader Crisis trigger: Crisis currently triggers only on a card-driven dip into ≤ 4. Should it also trigger if credit STARTS ≤ 4 in a save-loaded game? Defensive — probably not worth it.
  • v0.19 — Crisis-aware card behavior: Some cards could check s.flags.creditCrisisFired and behave differently. Example: Anti-Fed Pamphlet payment scaled higher post-Crisis (revenue collection harder during financial crisis). This is balance-shaping territory; not a failure-pressure question.
  • v0.19 — second Crisis penalty option: If +1 Resistance proves too anemic in playtesting, swap to "all players pay 25 TN" (single, mild). Don't compound — too easy to overshoot non-terminal.
  • v0.19 alt — chain Crisis to a card draw: "After Credit Crisis fires, the next Republic Debate draws Anti-Federalist Pamphlet (forced)." Mechanically links the warning state to additional pressure. Risk: removes Republic Debate variety.

None of these are authorized. Each would need its own explicit kickoff. The product reading is that v0.18 may be a stopping point: the failure-pressure system now has the right shape, and further iteration should be playtest-driven not sim-driven.
